# 接口定义（Data / Augment / Pose）

本文档用于约定 TrafficCopilot 项目中三项关键接口的“输入/输出/字段/约束”，以支持成员并行开发并确保后续可无缝集成：

- 训练数据接口（B ↔ D）：训练/验证/测试清单与类别契约
- 数据增强接口（D → B）：Albumentations 增强器的调用契约
- 骨架提取接口（E → B）：MediaPipe Pose 的向量化输出契约

---

## 0. 版本与术语

### 0.1 版本信息

- 接口版本：v1.1
- 生效范围：训练与评估流程、融合训练、演示推理
- 修改原则：任何字段变更必须更新本文件版本号，并同步修改数据生成脚本与训练脚本

### 0.2 术语与约定

- **项目根目录**：`D:\my_projects\TrafficCopilot`
- **相对路径**：均以项目根目录为基准（跨机器可复现）
- **split**：`train` / `valid` / `test`（以 Roboflow 导出目录为准）
- **group_id**：同一视频/同一采集段/同一场景序列的分组标识，用于防止数据泄漏
- **label_raw**：数据源原始类别名称
- **label**：训练用目标类别名称（可能由 label_raw 映射/合并得到）
- **label_id**：`classes.txt` 中的行号（从 0 开始）

## 1. 训练数据接口（B ↔ D）

目标：统一“数据怎么读、标签怎么对齐、划分怎么复现”，避免 train/val/test 泄漏导致评估虚高。

### 1.1 目录与文件位置（现行实现）

当前训练与融合训练脚本使用 Roboflow 导出的数据组织方式作为“训练入口”，不依赖 `data/annotations/*.csv`：

```text
<dataset_root>/
├─ train/
│  ├─ _classes.csv
│  └─ *.jpg
├─ valid/
│  ├─ _classes.csv
│  └─ *.jpg
└─ test/
   ├─ _classes.csv
   └─ *.jpg
```

其中：
- `<dataset_root>` 在配置文件中通过 `data.dataset_root` 指定（例如 `demo/traffic gestures.v1i.multiclass`）
- `_classes.csv` 为 Roboflow 导出的多分类（one-hot）标注文件，训练脚本会解析成“单标签分类任务”

### 1.2 Roboflow `_classes.csv` 契约（现行必需）

文件位置：
- `<dataset_root>/<split>/_classes.csv`（split 为 `train/valid/test`）

格式要求：

- 第一列必须为 `filename`
- 后续每列为一个类别名（列名即 `label_raw`）
- 每一行只有一个类别列为 `1`（或最大值），其余为 `0`

解析规则（与代码一致）：
- `label_raw = argmax(one_hot_columns)`
- `label_index = class_to_index[label_raw]`（类别顺序由 CSV 表头决定）

```text
data/
├─ raw/                      # 原始数据（只读，不覆盖）
├─ annotations/              # 标注契约文件（训练消费入口）
│  ├─ classes.txt            # 训练类别顺序（label_id 契约）
│  ├─ class_mapping.json     # 原始类别 -> 训练类别（可选，但推荐）
│  ├─ train.csv              # 训练清单（CSV 契约）
│  ├─ val.csv                # 验证清单（CSV 契约）
│  └─ test.csv               # 测试清单（CSV 契约）
└─ splits/
   └─ split_v1.json          # 可选：记录分组划分策略与统计信息
```

说明：
- 现行数据位于：`demo/traffic gestures.v1i.multiclass/`
- 后续若要做“更强可复现的数据治理”（分组划分、防泄漏、类别映射），再引入 `data/annotations/*.csv` 作为“标准入口”（见下一节“推荐增强”）

### 1.3 `classes.txt` / `class_mapping.json` / `train.csv`（推荐增强，非现行必需）

当你们需要把 8 类映射/合并为更贴合系统语义的训练类别集合时（例如 `wait for left turn` 与 `turn left` 的处理），建议新增 `data/annotations/` 契约文件，将“类别顺序”与“类别映射”从 Roboflow CSV 表头中解耦出来：

格式：

```text
<label_0>
<label_1>
...
<label_N-1>
```

约束：
- 不允许空行
- 不允许重复类别名
- 类别名建议使用小写下划线风格（如 `go_straight`），便于跨模块一致

#### 1.3.1 `classes.txt`

`data/annotations/classes.txt` 每行一个训练类别名，行号为 `label_id`。训练/推理/评估必须使用同一份文件，禁止在代码中硬编码类别顺序。

约束：
- 不允许空行
- 不允许重复类别名

#### 1.3.2 `class_mapping.json`

用于将数据源原始类别（label_raw）映射到训练类别（label）。当存在类别合并时必须提供此文件。

文件：`data/annotations/class_mapping.json`

格式：

```json
{
  "<label_raw_1>": "<label>",
  "<label_raw_2>": "<label>"
}
```

约束：
- 所有 `label_raw` 必须能映射到 `classes.txt` 中的某个 label
- 若出现未知标签，数据准备脚本应直接报错（避免静默丢数据）

#### 1.3.3 `train.csv` / `val.csv` / `test.csv`

文件（推荐）：
- `data/annotations/train.csv`
- `data/annotations/val.csv`（注意：这里使用 `val` 命名仅作为清单文件习惯，不影响数据目录 `valid/`）
- `data/annotations/test.csv`

字段定义（必需字段）：

| 列名 | 类型 | 示例 | 说明 |
|---|---|---|---|
| `id` | string | `frame_2865_8127e4` | 样本唯一标识，用于缓存与追踪（建议由文件名或 hash 生成） |
| `image_path` | string | `data/raw/roboflow/train/img.jpg` | 相对项目根目录路径 |
| `label_raw` | string | `wait for  left turn` | 数据源原始类别名 |
| `label` | string | `turn_left` | 训练用类别名（可由映射表产生） |
| `source` | string | `roboflow_v1` | 数据来源标记（便于统计与消融） |
| `group_id` | string | `video_01` | 分组 ID：同一视频/同一采集段必须同组（防泄漏关键） |

可选字段（建议保留）：

| 列名 | 类型 | 示例 | 说明 |
|---|---|---|---|
| `frame_index` | int | `2865` | 若为视频抽帧，可存帧序号，便于分析 |
| `width` / `height` | int | `640` | 原图尺寸（可选） |
| `note` | string | `night` | 备注（夜间/雨雾/遮挡等） |

约束与规则（建议）：
- 若数据来自视频抽帧，必须实现 `group_id` 的分组划分，避免同一视频相邻帧泄漏到不同 split
- 若存在类别合并，必须提供 `class_mapping.json` 并做一致性检查

### 1.4 训练侧（B）消费规则（现行强约束）

B 的训练脚本在数据层只做以下固定操作：

现行训练脚本（`scripts/train_mobilenetv3.py`）的消费规则为：

1. 从 `<dataset_root>/<split>/_classes.csv` 读取类别表头，构建 `class_names` 与 `class_to_index`
2. 对每行样本解析 `label_raw`，得到 `label_index`
3. 加载 `<dataset_root>/<split>/<filename>` 对应图片并执行 transform

补充：
- 现行实现的 `label_id` 顺序由 Roboflow CSV 表头决定；若未来要做类别合并/统一顺序，应切换到 `data/annotations/classes.txt` 契约

禁止事项：
- 禁止在训练代码中硬编码类别顺序
- 禁止在训练代码中自作主张丢弃“未知 label”，必须抛错

## 2. 数据增强接口（D → B）

目标：让 B 的 Dataset 只需“调用增强器”，不关心 Albumentations 的实现细节；并确保增强效果可配置、可复现。

### 2.1 输入输出数据类型（现行强约束）

现行训练代码的 transform 契约为“可调用对象”：

- 输入：`PIL.Image.Image`（RGB）
- 输出：`torch.Tensor`（`C x H x W`，float32），且已经完成 `Normalize(mean,std)`

当前实现支持两种后端（由 `augment.backend` 配置控制）：

- `torchvision`：`torchvision.transforms.Compose`，输入 PIL，输出 Tensor
- `albumentations`：`AlbumentationsWrapper(A.Compose(..., ToTensorV2()))`，输入 PIL（内部转 numpy），输出 Tensor

因此，增强接口应以“transform callable”的输入输出为准，而不是以 numpy uint8 为对外契约。

### 2.2 接口定义（现行函数式契约）

现行入口位于训练脚本内部（`scripts/train_mobilenetv3.py`）：

- `build_torchvision_transforms(config) -> (train_tf, eval_tf)`
- `build_albumentations_transforms(config) -> (train_tf, eval_tf)`
- `build_transforms(config) -> (train_tf, eval_tf)`

其中 `train_tf/eval_tf` 必须满足：

| 项 | 输入 | 输出 |
|---|---|---|
| `train_tf` | PIL RGB | Tensor float32（Normalize 后） |
| `eval_tf` | PIL RGB | Tensor float32（Normalize 后） |

### 2.3 增强策略与约束（现行实现对齐）

增强目标：覆盖真实道路干扰，但不能破坏“手势判别关键区域”（尤其是手臂姿态）。

建议增强类别：

- 光照与色彩：亮度、对比度、饱和度、色偏
- 天气与退化：雾、雨、噪声、压缩伪影
- 模糊：运动模糊、轻度高斯模糊
- 遮挡：小面积随机遮挡（避免遮掉整条手臂）
- 几何：小角度旋转、轻度缩放平移（避免强裁剪）

强约束（与当前配置一致）：
- `train` 使用随机增强；`valid/test` 仅做 Resize + CenterCrop + Normalize
- 增强强度与概率全部来自配置文件（例如 `config/mobilenetv3_albu_weather.yaml`）

### 2.4 B 侧使用规则（固定逻辑）

- `split == train`：调用 `train_augment`
- `split != train`：调用 `eval_augment`

如果 D 暂时未完成增强模块，B 可先用“空增强器”跑通训练（返回原图）。

## 3. 骨架提取接口（E → B）

目标：把 MediaPipe Pose 的关键点输出转成“固定长度向量 + 质量标记”，确保融合训练/推理可稳定对接。

### 3.1 输入输出（现行强约束）

现行实现采用“离线提取 → JSON 文件 → 训练读取”的接口方式：

- 提取脚本：`scripts/extract_pose_keypoints.py`
- 输出目录（默认）：`output/pose_keypoints/`
- 输出文件：`train_pose_keypoints.json`、`valid_pose_keypoints.json`、`test_pose_keypoints.json`

提取输入：
- 数据集图片：从 `<dataset_root>/<split>/*` 加载（同训练数据接口）

提取输出：每个 split 一个 JSON 数组，每个元素为一条 record（详见下一节 JSON Schema）。

### 3.2 JSON Schema（现行必需）

`<split>_pose_keypoints.json` 为数组，元素 record 必须包含以下字段（与脚本输出一致）：

| 字段 | 类型 | 说明 |
|---|---|---|
| `image_path` | string | 相对项目根目录路径（用于索引匹配） |
| `label_index` | int | 类别 id（由 `_classes.csv` 表头顺序决定） |
| `label_name` | string | 原始类别名（来自 `_classes.csv` 表头） |
| `pose_detected` | bool | 是否检测到 pose |
| `num_poses` | int | 检测到的 pose 数量 |
| `landmarks` | array | 第一条 pose 的 33 个关键点（若无则空数组） |
| `poses` | array | 所有 pose 的关键点集合（通常取第 1 条即可） |

单个 landmark 的字段（与脚本输出一致）：

| 字段 | 类型 | 说明 |
|---|---|---|
| `x,y,z` | float | 归一化坐标（0~1 为主，z 为相对深度） |
| `visibility` | float | 可见性/置信度（0~1） |
| `presence` | float | presence（0~1） |

### 3.3 `pose_vector` 特征维度与提取规则（现行实现）

融合训练不直接使用 landmarks，而是使用 `core/utils/pose_features.py` 将 record 转为固定维度特征向量：

- 函数：`extract_pose_feature_vector(record) -> np.ndarray(float32)`
- 现行维度：52（缺失 landmarks 时返回全 0 向量）

特征构成（与实现一致）：
- 上半身与躯干 8 点（肩/肘/腕/髋）的归一化坐标（x,y,z）+ visibility
- 若干上肢向量特征（含向量长度）
- 全局统计量（肩宽、躯干高度等）

v1 推荐仅使用上肢 6 点（保证稳定与任务相关性）：

- 左肩、右肩、左肘、右肘、左腕、右腕

向量构造：

- 取每点的 (x, y)，共 6×2 = 12 维
- **相对坐标**：以肩中心为原点
- **尺度归一**：除以肩宽（两肩距离），抵抗远近变化

最终：
- `pose_vector.shape == (12,)`
- dtype 为 float32

### 3.4 `pose_score` / `pose_ok`（建议补充项）

现行 JSON record 已包含 `pose_detected`，但缺少可用于“融合门控”的质量分数。建议后续在 record 中新增：

- `pose_score`：基于关键点 visibility 的均值或下四分位数
- `pose_ok`：`pose_score >= threshold` 的布尔值

以支持：
- 训练阶段：对低质量 pose 样本做降权/不融合
- 推理阶段：pose 质量差时降级为纯图像分支，提升安全性与稳定性

### 3.4 缓存接口（推荐，便于训练加速与复现）

建议提供离线缓存文件（按 split）：

- `data/processed/pose_cache/train.npz`
- `data/processed/pose_cache/val.npz`
- `data/processed/pose_cache/test.npz`

缓存键必须与 `train.csv` 的 `id` 一致，避免路径变化导致缓存失效。

缓存内容建议：

- `ids`: string array
- `pose_vectors`: float32 matrix `[N, 12]`
- `pose_scores`: float32 array `[N]`
- `pose_ok`: bool array `[N]`

### 3.5 B 侧消费规则（固定逻辑）

融合训练推荐逻辑：

- 训练时：可优先读取缓存；无缓存则在线提取（或要求先跑缓存脚本）
- 推理时：若 `pose_ok == False`，直接降级纯图像分支或降低输出置信度

## 4. 交付验收清单（对齐“能集成”的标准）

### 4.1 数据接口（B↔D）验收（现行）

- `<dataset_root>/train|valid|test/_classes.csv` 存在且可读
- CSV 表头类别名在三个 split 中一致（避免 train/valid/test 类别列不一致）
- 每条样本能解析出唯一 `label_name`，对应图片文件存在

### 4.2 增强接口（D→B）验收（现行）

- `augment.backend` 可在 `torchvision` 与 `albumentations` 间切换并成功运行
- `train_tf/eval_tf` 输入 PIL、输出 Tensor，且维度与 normalize 一致
- 提供 20 张增强前后对比图（用于人工检查增强是否破坏手势）

### 4.3 Pose 接口（E→B）验收（现行）

- `output/pose_keypoints/<split>_pose_keypoints.json` 生成成功且含必要字段
- `core/utils/pose_features.extract_pose_feature_vector` 输出维度固定（现行为 52）
- `scripts/train_fusion.py` 能加载 pose JSON 并完成一次完整训练/验证流程

## 5. 视频管线接口（交警定位 → ROI → 分类 → 时序意图）

目标：在连续视频中先定位“主要指挥者/交警”（bbox + track），再对 ROI 执行分类与时序解析，保证后续模块对真实路口输入具有一致的入口。

### 5.1 交警定位（Detector）

- 输入：`frame_bgr`（OpenCV 读入的 BGR 帧）+ `timestamp_ms`
- 输出：`detections[]`

字段约束：
- `bbox`：`[x1, y1, x2, y2]`（像素坐标，必须 clip 到帧内）
- `score`：0~1，代表“该 bbox 为主要指挥者”的可信度

### 5.2 跟踪（Tracker）

- 输入：`detections[]`
- 输出：`tracks[]`（至少包含 `track_id/bbox/score/age/lost`）

行为约束：
- 允许短时丢失维持上一帧 bbox（通过 `lost` 计数）
- `lost > max_lost` 必须清空轨迹（进入重新发现）

### 5.3 ROI 裁剪（Crop）

- 输入：`tracks[0].bbox`
- 输出：ROI 图像（PIL RGB）

约束：
- ROI 为空必须跳过本帧预测（避免异常）
- 颜色空间统一为 RGB，避免 BGR/RGB 混淆

### 5.4 分类与意图

- 分类：沿用 `Predictor.predict(image)->Prediction`
- 意图：沿用 `IntentEngine.update(prediction_dict)->intent_dict`
