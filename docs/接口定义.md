# 接口定义（Data / Augment / Pose）

本文档用于约定 TrafficCopilot 项目中三项关键接口的“输入/输出/字段/约束”，以支持成员并行开发并确保后续可无缝集成：

- 训练数据接口（B ↔ D）：训练/验证/测试清单与类别契约
- 数据增强接口（D → B）：Albumentations 增强器的调用契约
- 骨架提取接口（E → B）：MediaPipe Pose 的向量化输出契约

---

## 0. 版本与术语

### 0.1 版本信息

- 接口版本：v1
- 生效范围：训练与评估流程、融合训练、演示推理
- 修改原则：任何字段变更必须更新本文件版本号，并同步修改数据生成脚本与训练脚本

### 0.2 术语与约定

- **项目根目录**：`D:\my_projects\TrafficCopilot`
- **相对路径**：均以项目根目录为基准（跨机器可复现）
- **split**：`train` / `val` / `test`
- **group_id**：同一视频/同一采集段/同一场景序列的分组标识，用于防止数据泄漏
- **label_raw**：数据源原始类别名称
- **label**：训练用目标类别名称（可能由 label_raw 映射/合并得到）
- **label_id**：`classes.txt` 中的行号（从 0 开始）

## 1. 训练数据接口（B ↔ D）

目标：统一“数据怎么读、标签怎么对齐、划分怎么复现”，避免 train/val/test 泄漏导致评估虚高。

### 1.1 目录与文件位置（推荐）

```text
data/
├─ raw/                      # 原始数据（只读，不覆盖）
├─ annotations/              # 标注契约文件（训练消费入口）
│  ├─ classes.txt            # 训练类别顺序（label_id 契约）
│  ├─ class_mapping.json     # 原始类别 -> 训练类别（可选，但推荐）
│  ├─ train.csv              # 训练清单（CSV 契约）
│  ├─ val.csv                # 验证清单（CSV 契约）
│  └─ test.csv               # 测试清单（CSV 契约）
└─ splits/
   └─ split_v1.json          # 可选：记录分组划分策略与统计信息
```

说明：
- 目前仓库已有 Roboflow 数据集位于 `demo/traffic gestures.v1i.multiclass/`。后续建议复制或迁移到 `data/raw/`，并由脚本生成 `data/annotations/*.csv` 作为训练入口。

### 1.2 `classes.txt`（必需）

`data/annotations/classes.txt` 每行一个训练类别名，行号为 `label_id`。训练/推理/评估必须使用同一份文件，禁止在代码中硬编码类别顺序。

格式：

```text
<label_0>
<label_1>
...
<label_N-1>
```

约束：
- 不允许空行
- 不允许重复类别名
- 类别名建议使用小写下划线风格（如 `go_straight`），便于跨模块一致

### 1.3 `class_mapping.json`（推荐）

用于将数据源原始类别（label_raw）映射到训练类别（label）。当存在类别合并时必须提供此文件。

文件：`data/annotations/class_mapping.json`

格式：

```json
{
  "<label_raw_1>": "<label>",
  "<label_raw_2>": "<label>"
}
```

约束：
- 所有 `label_raw` 必须能映射到 `classes.txt` 中的某个 label
- 若出现未知标签，数据准备脚本应直接报错（避免静默丢数据）

### 1.4 `train.csv` / `val.csv` / `test.csv`（必需）

三个清单文件的字段必须一致。推荐 UTF-8 编码。

文件：
- `data/annotations/train.csv`
- `data/annotations/val.csv`
- `data/annotations/test.csv`

字段定义（必需字段）：

| 列名 | 类型 | 示例 | 说明 |
|---|---|---|---|
| `id` | string | `frame_2865_8127e4` | 样本唯一标识，用于缓存与追踪（建议由文件名或 hash 生成） |
| `image_path` | string | `data/raw/roboflow/train/img.jpg` | 相对项目根目录路径 |
| `label_raw` | string | `wait for  left turn` | 数据源原始类别名 |
| `label` | string | `turn_left` | 训练用类别名（可由映射表产生） |
| `source` | string | `roboflow_v1` | 数据来源标记（便于统计与消融） |
| `group_id` | string | `video_01` | 分组 ID：同一视频/同一采集段必须同组（防泄漏关键） |

可选字段（建议保留）：

| 列名 | 类型 | 示例 | 说明 |
|---|---|---|---|
| `frame_index` | int | `2865` | 若为视频抽帧，可存帧序号，便于分析 |
| `width` / `height` | int | `640` | 原图尺寸（可选） |
| `note` | string | `night` | 备注（夜间/雨雾/遮挡等） |

约束与规则：
- 训练侧只能消费 CSV（不要直接读原始 `*_classes.csv`），保证统一入口
- 必须保证同一 `group_id` 只出现在一个 split 中（强制防泄漏）
- 若 `label` 不在 `classes.txt` 中，数据准备脚本必须报错

### 1.5 训练侧（B）消费规则（强约束）

B 的训练脚本在数据层只做以下固定操作：

1. 读取 `classes.txt`，构建 `label -> label_id`
2. 读取 `train.csv/val.csv/test.csv`，构建样本列表
3. `image_path` 加载图像，取 `label` 转换为 `label_id`

禁止事项：
- 禁止在训练代码中硬编码类别顺序
- 禁止在训练代码中自作主张丢弃“未知 label”，必须抛错

## 2. 数据增强接口（D → B）

目标：让 B 的 Dataset 只需“调用增强器”，不关心 Albumentations 的实现细节；并确保增强效果可配置、可复现。

### 2.1 输入输出数据类型（强约束）

输入图像：
- 类型：`numpy.ndarray`
- 形状：`H x W x 3`
- 颜色空间：`RGB`
- dtype：`uint8`（0~255）

输出图像：
- 与输入同约束：`H x W x 3`、`RGB`、`uint8`

说明：
- Resize 与 normalize 既可放在 Albumentations 中，也可放在 Dataset 末端，但必须保证训练与推理一致。

### 2.2 接口定义（函数式契约）

D 对外提供两个函数（或等价的可调用对象）：

- `build_train_augment(config) -> Callable`
- `build_eval_augment(config) -> Callable`

调用约定：

```text
aug = build_train_augment(config)
out = aug(image_rgb_uint8)
```

返回值 `out` 必须为增强后的 `image_rgb_uint8`（或包含 `image` 字段的 dict，但需固定约定；建议直接返回 image，减少 B 侧适配工作）。

### 2.3 增强策略与约束（推荐）

增强目标：覆盖真实道路干扰，但不能破坏“手势判别关键区域”（尤其是手臂姿态）。

建议增强类别：

- 光照与色彩：亮度、对比度、饱和度、色偏
- 天气与退化：雾、雨、噪声、压缩伪影
- 模糊：运动模糊、轻度高斯模糊
- 遮挡：小面积随机遮挡（避免遮掉整条手臂）
- 几何：小角度旋转、轻度缩放平移（避免强裁剪）

强约束：
- 只对 `train` 使用增强；`val/test` 禁止使用随机增强（仅做必要的 resize/normalize）
- 增强强度必须可配置（建议写入 `config/augment.yaml`），以支持消融与复现实验

### 2.4 B 侧使用规则（固定逻辑）

- `split == train`：调用 `train_augment`
- `split != train`：调用 `eval_augment`

如果 D 暂时未完成增强模块，B 可先用“空增强器”跑通训练（返回原图）。

## 3. 骨架提取接口（E → B）

目标：把 MediaPipe Pose 的关键点输出转成“固定长度向量 + 质量标记”，确保融合训练/推理可稳定对接。

### 3.1 输入输出（强约束）

输入：
- `image_rgb_uint8`：`H x W x 3`、RGB、uint8
- 可选：`roi_bbox`：`[x1, y1, x2, y2]`（像素坐标，左上/右下），用于对 ROI 内做人像 pose（推荐最终集成）

输出（必须包含 3 个字段）：

| 字段 | 类型 | 说明 |
|---|---|---|
| `pose_vector` | `np.ndarray(float32)` | 固定维度向量（v1 推荐 12 维） |
| `pose_score` | `float` | 0~1 整体可信度（由关键点 visibility/置信度汇总） |
| `pose_ok` | `bool` | 是否可用（`pose_score` 超阈值且点集完整） |

### 3.2 `pose_vector` 固定维度定义（v1 推荐）

v1 推荐仅使用上肢 6 点（保证稳定与任务相关性）：

- 左肩、右肩、左肘、右肘、左腕、右腕

向量构造：

- 取每点的 (x, y)，共 6×2 = 12 维
- **相对坐标**：以肩中心为原点
- **尺度归一**：除以肩宽（两肩距离），抵抗远近变化

最终：
- `pose_vector.shape == (12,)`
- dtype 为 float32

### 3.3 `pose_score` 与 `pose_ok` 规则（v1 推荐）

建议计算方式：

- `pose_score = 上述 6 个点的 visibility（或置信度）均值`
- `pose_ok = (6 点齐全) AND (pose_score >= 0.60)`

降级逻辑（由 B 在融合时实现，但需遵守接口语义）：

- 若 `pose_ok == False`：融合模型可用零向量替代 `pose_vector`，或直接走纯图像分支（推荐推理阶段）

### 3.4 缓存接口（推荐，便于训练加速与复现）

建议提供离线缓存文件（按 split）：

- `data/processed/pose_cache/train.npz`
- `data/processed/pose_cache/val.npz`
- `data/processed/pose_cache/test.npz`

缓存键必须与 `train.csv` 的 `id` 一致，避免路径变化导致缓存失效。

缓存内容建议：

- `ids`: string array
- `pose_vectors`: float32 matrix `[N, 12]`
- `pose_scores`: float32 array `[N]`
- `pose_ok`: bool array `[N]`

### 3.5 B 侧消费规则（固定逻辑）

融合训练推荐逻辑：

- 训练时：可优先读取缓存；无缓存则在线提取（或要求先跑缓存脚本）
- 推理时：若 `pose_ok == False`，直接降级纯图像分支或降低输出置信度

## 4. 交付验收清单（对齐“能集成”的标准）

### 4.1 数据接口（B↔D）验收

- `classes.txt` 存在且无重复、无空行
- `train/val/test.csv` 列齐全、可被脚本读取
- `group_id` 无跨 split 重复（需自动检查并给出报告）
- 任意一条 `label` 都能在 `classes.txt` 中找到

### 4.2 增强接口（D→B）验收

- `build_train_augment` / `build_eval_augment` 可被导入并调用
- 输入输出类型符合 RGB uint8 约束
- 提供 20 张增强前后对比图（用于人工检查增强是否破坏手势）

### 4.3 Pose 接口（E→B）验收

- `extract_pose(image, roi_bbox=None)` 输出包含 `pose_vector/pose_score/pose_ok`
- `pose_vector` 固定为 12 维 float32
- 提供关键点叠加可视化样例（用于验证提取有效性）
