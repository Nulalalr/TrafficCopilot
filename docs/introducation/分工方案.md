# TrafficCopilot 项目详细分工方案

## 1. 分工原则与目标

为充分体现团队协作与技术深度，本方案遵循以下原则：

- **人人皆核心**：确保每位成员深度参与至少一项关键算法或工程模块的研发，避免边缘化工作。
- **模块解耦与协作**：将系统拆解为五个相对独立又相互依赖的核心模块，通过清晰的数据接口进行集成。
- **工作量均衡**：各成员预期投入时间相当，队长额外承担统筹与展示职责，工作量略高。
- **成果可视化**：每位成员需交付可量化、可展示的技术产出，便于在答辩与报告中独立阐述贡献。

## 3. 各成员详细任务说明

### 3.1 队长 A：系统架构与意图推理（统筹岗）

**职责概述**：负责项目总体技术路线规划、核心创新点的算法逻辑设计、以及最终成果的整合与展示。不要求编写大量重复性代码，但需产出指导团队方向的顶层设计。

**具体技术任务**：

| 序号 | 任务描述 | 技术细节 | 预计耗时 |
| :--- | :--- | :--- | :--- |
| 1 | 系统总体架构设计 | 绘制清晰的数据流图与控制流图，定义各模块间的数据接口（输入/输出格式、尺寸、频率）。 | 8h |
| 2 | 连续手势意图解析器设计 | 针对单帧识别结果，设计基于**滑动窗口 + 有限状态机 (FSM)** 的时序融合模块。定义状态转移条件（例如：连续 N 帧识别为“停止”才判定为“停止意图”，单帧噪声不触发指令变化）。 | 12h |
| 3 | 车辆控制指令映射表制定 | 将高层意图（如“左转待转”）映射为标准化的车辆控制域信号（如 CAN 总线信号定义建议），撰写《接口控制文件 (ICD) 草案》。 | 6h |
| 4 | 项目进度跟踪与风险管理 | 使用 GitHub Projects 或飞书表格管理成员任务状态，每周同步。 | 持续 |
| 5 | 答辩 PPT 制作与路演准备 | 将技术路线转化为具有故事性的演示文稿，突出创新性与应用价值。 | 10h |

**交付物清单**：
- `docs/architecture_diagram.pdf`：系统整体架构图。
- `core/intent_fsm.py`：意图状态机的 Python 伪代码/框架代码。
- `docs/ICD_interface.pdf`：车辆控制接口定义文档。
- `presentation/TrafficCopilot_答辩PPT.pptx`：最终答辩幻灯片。

### 3.2 队员 B：视觉特征提取网络设计与改进

**职责概述**：负责深度学习模型的骨干网络选型、自定义模块开发与训练，确保模型对手势区域具有高敏感性。

**具体技术任务**：

| 序号 | 任务描述 | 技术细节 | 预计耗时 |
| :--- | :--- | :--- | :--- |
| 1 | 基线模型搭建与对比 | 基于 PyTorch 搭建 MobileNetV4、EfficientNet-B0 等轻量级分类网络，完成初始训练与准确率基准测试。 | 10h |
| 2 | 注意力机制引入与改进 | 在骨干网络中插入**坐标注意力 (Coordinate Attention)** 模块。修改网络定义文件，使其能输出特征热力图用于可视化分析。 | 15h |
| 3 | 模型训练与消融实验 | 在同一数据集上训练改进前后的模型，记录训练曲线，生成 Grad-CAM 热力图对比，撰写实验分析段落。 | 12h |
| 4 | 模型权重保存与交接 | 将训练好的 `best_model.pth` 及网络配置文件提交给队员 C 进行轻量化处理。 | 2h |

**交付物清单**：
- `core/model/backbone.py`：包含注意力机制的改进网络代码。
- `experiments/ablation_study/`：消融实验对比图（准确率曲线、热力图）。
- `docs/technical_report_model_improvement.md`：网络改进技术说明文档。

### 3.3 队员 C：模型轻量化与边缘部署

**职责概述**：负责将队员 B 训练的高精度模型压缩并适配至边缘计算平台，以满足车规级的实时性要求。

**具体技术任务**：

| 序号 | 任务描述 | 技术细节 | 预计耗时 |
| :--- | :--- | :--- | :--- |
| 1 | 模型导出与转换 | 将 `.pth` 权重文件导出为 **ONNX** 通用格式，并进行模型结构简化（如算子融合）。 | 6h |
| 2 | 量化与剪枝优化 | 使用 NVIDIA TensorRT 或 PyTorch 原生量化工具对模型进行 **INT8 量化**，或采用结构化剪枝减少参数量。 | 14h |
| 3 | 推理性能 Benchmark | 在笔记本 GPU 环境（模拟 Jetson Orin）下测试推理速度 (FPS) 与内存占用，制作性能对比表格。 | 8h |
| 4 | 编写部署接口脚本 | 提供 `deploy/inference.py`，封装模型加载与推理函数，确保队员 D 和队员 E 能直接调用。 | 6h |

**交付物清单**：
- `weights/best_model_int8.onnx`：最终可部署的量化模型。
- `deploy/benchmark_report.xlsx`：不同优化策略下的延迟与吞吐量数据。
- `deploy/inference.py`：用于集成的推理接口脚本。

### 3.4 队员 D：鲁棒性数据增强与评估体系

**职责概述**：负责构建恶劣环境下的测试数据集，并建立科学的评估体系，量化系统在不同干扰下的性能衰减，证明系统的“高鲁棒性”。

**具体技术任务**：

| 序号 | 任务描述 | 技术细节 | 预计耗时 |
| :--- | :--- | :--- | :--- |
| 1 | 自动化数据增强流水线开发 | 基于 **Albumentations** 库编写脚本，模拟雨滴、雾霾、强光过曝、运动模糊、椒盐噪声等复杂场景。 | 10h |
| 2 | 评估指标脚本编写 | 编写 Python 脚本计算 **准确率、召回率、F1-Score、混淆矩阵**，并自动生成可视化图表。 | 10h |
| 3 | 鲁棒性对比测试 | 在原始干净测试集与增强后的退化测试集上分别运行模型，计算性能保持率。 | 8h |
| 4 | 硬件在环 (HIL) 仿真方案设计 | 撰写《ROS2 节点集成方案》，设计如何将本模块的输出以 `TrafficGesture.msg` 话题形式发布给车辆决策模块。 | 6h |

**交付物清单**：
- `core/utils/augmentation.py`：复杂环境数据增强工具包。
- `scripts/evaluate.py`：自动化测试与评估脚本。
- `docs/robustness_evaluation_report.pdf`：鲁棒性评估报告（含雷达图）。
- `docs/HIL_integration_design.md`：硬件在环仿真方案设计文档。

**职责概述**：通过引入人体骨架关键点作为辅助信息提升识别精度，并负责开发面向演示和用户体验的交互界面。

**交付物清单**：

- `demo/app.py`：Gradio 交互式演示程序。
- `demo/demo_video.mp4`：项目功能演示视频。

## 4. 协作流程与接口定义

为确保各模块无缝集成，定义以下关键数据接口：

| 上游模块（提供者） | 下游模块（消费者） | 接口内容 | 格式/协议 |
| :--- | :--- | :--- | :--- |
| 队员 D (数据) | 队员 B (训练) | 增强后的训练图片与标注 | `train_images/`, `labels.json` |
| 队员 B (模型) | 队员 C (部署) | 训练完成的模型权重 | `model.pth` (PyTorch) |
| 队员 C (部署) | 队员 D/E (测试/演示) | 推理函数接口 | `def predict(img_path) -> (label, conf)` |
| 队长 A (逻辑) | 队员 D/E (集成) | 时序意图解码器函数 | `def decode_intent(frame_list) -> intent` |

**协作沟通机制**：
- **每日站会 (10分钟)**：钉钉/飞书群内文字同步昨日进展、今日计划。
- **每周周会 (30分钟)**：全员线上会议，解决集成卡点问题。
- **代码管理**：使用 GitHub 仓库，各成员维护自己的 `feature/xxx` 分支，合并前需提交 Pull Request 并由队长 Review。

## 6. 风险预案

- **风险 1：模型准确率不达标**  
  预案：队员 B 与队员 E 重点攻关数据增强策略；队员 D 增加针对性难例挖掘。

- **风险 2：轻量化后精度损失过大**  
  预案：队员 C 采用 **量化感知训练 (QAT)** 替代训练后量化，并与队员 B 协同微调。

- **风险 3：集成接口不匹配**  
  预案：队长 A 在第二周结束前确认最终接口定义文档，避免返工。
