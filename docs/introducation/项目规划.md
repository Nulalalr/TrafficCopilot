# TrafficCopilot 项目开发规划书（周计划版）

## 1. 项目总体时间线与阶段划分

本项目计划总工期 **4 周**，分为五个阶段：

| 阶段 | 周期 | 阶段目标 |
| :--- | :--- | :--- |
| 阶段 0 | 项目启动前 | 环境统一、数据准备、分工确认 |
| 阶段 1 | 第 1 周 | 基线模型跑通、各模块框架搭建 |
| 阶段 2 | 第 2 周 | 核心算法研发并行推进 |
| 阶段 3 | 第 3 周 | 模块集成、系统联调与评估 |
| 阶段 4 | 第 4 周 | 文档撰写、演示制作与最终交付 |

---

## 2. 每周详细任务规划（并发分工）

### 📅 阶段 0：项目启动前（第 0 周）

**目标**：统一开发环境、获取基础数据、明确接口定义。

| 成员 | 具体任务 | 产出物 | 与其他成员的交互 |
| :--- | :--- | :--- | :--- |
| **队长 A** | 1. 绘制系统整体架构草图（数据流+控制流）<br>2. 定义各模块间的数据接口格式<br>3. 建立 GitHub 仓库并设置分支保护规则 | `架构草图.png`<br>`接口定义文档.md`<br>GitHub 仓库结构 | 向全员宣讲架构，确认接口无歧义 |
| **队员 B** | 1. 搭建 PyTorch 训练环境<br>2. 下载 HaGRID 等开源手势数据集<br>3. 编写基线模型（MobileNetV3）的训练脚本框架 | `environment.yml`<br>数据集目录结构<br>`train_baseline.py` | 与 C 确认最终导出格式要求 |
| **队员 C** | 1. 配置 ONNX/TensorRT 转换环境<br>2. 编写模型转换与 Benchmark 测试框架<br>3. 调研轻量化方案（量化/剪枝）可行性 | `convert_onnx.py`<br>`benchmark.py` | 与 B 对齐模型输入输出尺寸 |
| **队员 D** | 1. 编写数据增强流水线原型（Albumentations）<br>2. 收集网络真实交警图片作为独立测试集雏形<br>3. 设计评估指标计算函数 | `augmentation.py`<br>测试图片 20 张<br>`metrics.py` | 与 E 共享测试图片用于演示 |
| **队员 E** | 1. 集成 MediaPipe 并测试关键点提取效果<br>2. 搭建 Gradio 空白应用框架<br>3. 设计演示界面的 UI 布局草图 | `extract_keypoints.py`<br>Gradio Hello World<br>UI 草图 | 与 B 讨论骨架特征如何传入模型 |

### 📅 阶段 1：基线搭建与模块骨架（第 1 周）

**目标**：每位成员的核心模块完成最小可行版本（MVP），跑通全流程。

| 成员 | 具体任务 | 预计耗时 | 产出物 | 协作依赖 |
| :--- | :--- | :--- | :--- | :--- |
| **队长 A** | 1. 完善意图状态机的伪代码逻辑<br>2. 编写一个模拟测试脚本，验证状态机对连续帧输入的输出稳定性<br>3. 组织第一次周会，解决卡点 | 6h | `fsm_intent.py`<br>周会纪要 | 向全员解释意图层如何接收模型输出 |
| **队员 B** | 1. 完成基线模型首轮训练（10 个 epoch）<br>2. 输出训练日志与初步准确率<br>3. 将模型权重保存并交给 C 进行转换测试 | 10h | `baseline_model.pth`<br>`training_log.csv` | 将权重交 C |
| **队员 C** | 1. 将 B 的基线模型转换为 ONNX 格式<br>2. 验证 ONNX 推理结果与 PyTorch 一致性<br>3. 运行首次 Benchmark，记录基线速度 | 8h | `baseline.onnx`<br>`speed_test.txt` | 接收 B 的权重 |
| **队员 D** | 1. 使用增强脚本对训练集进行离线增强并保存样本<br>2. 编写测试脚本，计算基线模型在干净测试集上的准确率（基准值） | 8h | 增强后样本文件夹<br>`baseline_accuracy.json` | 模型调用由 C 提供的推理接口 |
| **队员 E** | 1. 将关键点坐标归一化并保存为特征文件<br>2. 修改 Gradio 界面，实现单张图片上传并显示骨架叠加图 | 8h | `keypoints_cache.npy`<br>Gradio 骨架预览版 | 需 B 提供模型类别名称列表 |

### 📅 阶段 2：核心算法深化与优化（第 2 周）

**目标**：各成员深入自己负责的技术创新点，产出核心改进成果。

| 成员 | 具体任务 | 预计耗时 | 产出物 | 协作依赖 |
| :--- | :--- | :--- | :--- | :--- |
| **队长 A** | 1. 将状态机逻辑封装为独立 Python 类<br>2. 设计一个仿真数据生成器，模拟模型输出的连续帧置信度列表，测试意图解码准确率<br>3. 开始撰写 PPT 大纲与技术路线图 | 10h | `intent_decoder.py`<br>PPT 初版大纲 | 准备接收 C 的实时推理接口 |
| **队员 B** | 1. 在网络中插入坐标注意力模块（CA Block）<br>2. 重新训练改进网络，与基线对比<br>3. 生成 Grad-CAM 热力图，证明注意力集中在手臂区域 | 15h | `model_ca.pth`<br>热力图对比图 | 训练数据可使用 D 的增强集（可选） |
| **队员 C** | 1. 对 B 的改进模型进行 INT8 量化（或剪枝）<br>2. 编写推理脚本，支持批量图片输入<br>3. 优化推理 Pipeline，确保帧率达标 | 12h | `model_int8.onnx`<br>`inference_pipeline.py` | 接收 B 的新权重 |
| **队员 D** | 1. 在增强测试集（加雨、雾、模糊）上评估改进模型的性能衰减<br>2. 绘制鲁棒性雷达图<br>3. 设计硬件在环仿真的 ROS2 消息定义文档 | 10h | `robustness_report.png`<br>`TrafficGesture.msg` | 使用 C 的推理接口批量测试 |

## 5. 风险管理与应对

| 风险点 | 影响模块 | 应对预案 | 触发条件 |
| :--- | :--- | :--- | :--- |
| 改进模型训练不收敛 | B | 降低学习率，增加 warmup；换用预训练权重 | W2 中期 loss 不降 |
| 量化后精度大幅下降 | C | 采用量化感知训练 (QAT)；与 B 协同微调 | W2 末精度损失 > 5% |
| 融合模型特征维度错位 | E + B | 由队长 A 协调，召开专项调试会议 | W3 集成时报错 |
| 演示视频效果不佳 | E | 提前准备多组图片，设计好脚本 | W4 初版质量低 |
| 某成员进度落后 | 全员 | 每周周会暴露问题，队长调配其他成员支援非核心部分 | 连续两天无更新 |
