## 1. 项目简介（问题定义与价值）

### 1.1 谁在什么情况下遇到什么问题？

在真实城市道路中，驾驶主体（人类驾驶员、自动驾驶车辆）会在以下场景中遇到“交通规则临时切换”的问题：

- **信号灯故障/断电**：路口不再由固定信号控制，交通组织依赖现场交警临时指挥。
- **大型活动与临时管制**：道路封控、分流、潮汐车道调整需要人工指挥实时引导。
- **交通事故处置/道路施工**：车道占用、改道与单侧放行常由交警手势进行动态调度。

在这些情况下，交警的手势（如“停止”“直行”“左转待转”等）成为交通参与者必须遵从的高优先级指令。人类驾驶员可以凭经验快速理解并执行，但**当前多数无人驾驶系统仍难以稳定理解交警的动态手势意图**，从而存在误判、违规通行、指令抖动等风险，限制无人车在开放道路的全天候、全场景适用性。

### 1.2 为什么用视觉方法去解决？

交警手势属于典型的“视觉指令”，其核心信息主要体现在：

- 手臂/上肢的空间姿态与运动方向；
- 手势序列的时间结构（开始—保持—结束）；
- 场景语义（路口、车道、指挥方向）对意图的约束。

因此，视觉感知是最直接的解决路径：通过摄像头获取视频流，结合目标检测、姿态估计、手势分类、时序意图解析等视觉技术，将“人的动作”转换为“机器可执行的控制意图”。

### 1.3 已经有什么解决方法？

目前相关方向常见路线包括：

- **纯分类**：对单帧图片做手势分类，工程简单但易受噪声影响，输出抖动明显。
- **检测+分类**：先定位人体/手臂区域再分类，能提升背景鲁棒性，但仍缺乏时序理解。
- **姿态关键点驱动**：用人体关键点（骨架）表达上肢动作，结构先验更强，但在遮挡/远距离时容易丢点。
- **动作识别/时序模型**：引入滑动窗口、RNN/Transformer 等进行序列建模，性能更强但常带来算力与延迟压力。

### 1.4 我们的方案优势与用户价值

TrafficCopilot 的定位是无人驾驶平台可集成的“独立感知模块”，强调工程可用性与安全性。核心优势体现在：

- **闭环输出**：不仅输出手势类别，还输出稳定的“车辆控制域可理解指令”（含置信度与状态），更贴近实际系统接口需求。
- **鲁棒性导向**：设计面向复杂光照/雨雾/背景干扰的数据增强与评估体系，证明在恶劣工况下的性能保持率。
- **时序抗抖动**：通过滑动窗口融合 + 有限状态机（FSM）进行意图稳定化，降低单帧误判引起的指令抖动。
- **轻量可部署**：以车端边缘设备为约束，规划 ONNX/TensorRT 的部署链路，追求低延迟与可落地。

对用户（无人驾驶车辆/示范区运营方/安全评测方）的直接价值：

- 在信号灯失效、临时管控等场景下提升无人车遵从人工指挥的能力；
- 减少误判导致的违规通行与安全隐患，提升公众信任；
- 形成可复用的软件模块与评测方法，便于规模化推广与迭代。

## 2. 项目进度

### 2.1 技术路线图（文本版）

实践周期间我们以“能演示的最小可行产品（MVP）”为目标，先跑通端到端流程，再逐步替换为更强的模型与更完整的系统组件。

```text
摄像头/图片输入
   │
   ▼
图像预处理（尺寸缩放/归一化/质量评估）
   │
   ├───────────────┐
   ▼               ▼
人体/交警定位与ROI  骨架关键点提取（可选）
   │               │
   └───────┬───────┘
           ▼
手势分类（MVP：特征+KNN；后续：轻量深度模型）
           │
           ▼
时序意图解析（滑动窗口+FSM 抗抖动）
           │
           ▼
标准化指令输出（用于车辆控制/ROS2对接/日志）
```

- 深度学习与部署（规划/后续）：PyTorch、ONNX、TensorRT（面向边缘端）

### 2.3 数据来源与预处理方案

#### 2.3.1 当前已引入的数据

目前仓库中已包含一份 Roboflow 导出的多分类手势数据集（用于 MVP 与基线验证）：

- 路径：`demo/traffic gestures.v1i.multiclass/`
- 标注格式：Multi-Class Classification（每张图片一个类别）
- 类别（8类）：change lanes、go straight、pull over、slow down、stop、turn left、turn right、wait for left turn

数据集说明文件中标注了原始规模与导出处理方法（实践周阶段用于快速跑通流程）：  
- 原始图片约 267 张；导出时产生 3 个增强版本（亮度扰动、椒盐噪声）；统一缩放为 640×640（Stretch）。

#### 2.3.2 训练集构建与划分策略（建议）

为让后续深度模型训练更稳定，并符合“鲁棒性与实时性”目标，我们计划将训练数据整理到统一路径（按项目架构约定的 `data/` 目录），并执行如下策略：

- **类别映射与合并**：将 8 类映射为项目目标类别集合（例如将“wait for left turn”并入“turn left”或作为复合意图类）。
- **分层划分**：按场景维度划分 train/val/test（光照、天气、距离、遮挡），避免同一视频相邻帧泄漏到不同划分。
- **增强策略**：使用 Albumentations 生成雨雾、过曝、运动模糊、遮挡等退化条件样本，验证鲁棒性保持率。

### 3.2 MVP：端到端最小可行演示（可运行原型）

为满足实践周“能演示 MVP”的要求，我们优先实现可运行的最小闭环：**图片输入 → 特征提取 → 基线分类 → 时序意图稳定化 → Web 界面展示**。

对应实现文件（便于检查工作量）：  
- `demo/app.py`：包含数据加载、特征提取、KNN 分类器、时序意图引擎与 Web 接口。

MVP 的关键点：

- **特征提取**：以灰度缩放特征 + HOG 特征作为图像表示（实现了 HOG 的基础版本）。
- **分类器**：实现了基于距离的 KNN 分类，并输出 Top-k 结果、置信度与 margin。
- **时序意图**：实现了 `TemporalIntentEngine`，使用滑动窗口加权 + hold 策略抑制抖动（对连续帧输出更稳定）。
- **指令映射**：将数据集标签映射为标准化指令枚举（如 STOP、GO_STRAIGHT、TURN_LEFT 等），并提供中文解释。

说明：该 MVP 采用传统特征+KNN 作为快速验证手段，便于在没有深度学习训练环境的情况下完成端到端演示；后续将用轻量深度模型替换分类器模块。

## 4. 延期中（遇到的问题与分析）

### 4.1 深度学习训练与依赖环境尚未完全落地

当前仓库中 MVP 演示依赖 Flask、Pillow、NumPy；深度模型训练（PyTorch）与部署（ONNX/TensorRT）尚处于计划阶段。主要原因：

- 实践周时间有限，优先完成可演示闭环；
- 训练集规模与类别定义仍在收敛（8 类到目标类别的映射与合并方案需要进一步确定）；
- 需要在目标硬件或模拟环境上验证端到端延迟与吞吐，不适合在实践周短周期内一次完成。

### 4.2 数据规模偏小与场景偏差

现有数据主要来自开源数据集，存在：

- 真实“交警指挥”场景覆盖不足（夜间、雨雾、远距离、遮挡、复杂背景较少）；
- 类别分布可能不均衡，容易造成模型偏置；
- 若使用视频帧采样，可能出现相邻帧泄漏导致评估偏乐观。

### 4.3 错误分析与改进方向（实践周阶段结论）

我们预计主要错误将集中在：

- 外观相近类别之间混淆（例如 turn_left vs wait_for_left_turn；pull_over vs slow_down）；
- 低清晰度与模糊场景导致置信度下降；
- 单帧误判引起的指令抖动（需要靠时序稳定化抑制）。

## 5. 待开发与解决方案（下一步计划）

### 5.1 训练集完善（短期优先级最高）

解决方案：

1. 建立 `data/raw/`、`data/processed/`、`data/splits/` 的统一数据管理；
2. 明确最终类别体系与映射表（8类 → 目标类别/复合意图类别）；
3. 引入更大规模开源数据（如 HaGRID 等）并补充真实路口交警样本；
4. 使用 Albumentations 建立鲁棒性增强流水线，并形成“干净集/退化集”双评测集。

### 5.2 深度模型替换与轻量化部署（中期）

解决方案：

- 用轻量骨干（如 MobileNet 系列）替换 MVP 的 KNN 分类器，并引入注意力机制提升关键区域聚焦；
- 加入骨架关键点辅助分支，实现图像特征与结构特征融合；
- 完成 ONNX 导出与（可选）TensorRT 加速，评估端侧延迟与吞吐；
- 建立统一的推理接口，供评估脚本与演示系统调用。

### 5.3 场景化验证与评估闭环（中期到长期）

解决方案：

- 设计贴合路口场景的端到端用例：信号灯故障、事故封控、夜间雨雾；
- 指标体系覆盖：准确率/F1、鲁棒性保持率、端到端延迟、指令稳定性（切换频率/误触发率）；
