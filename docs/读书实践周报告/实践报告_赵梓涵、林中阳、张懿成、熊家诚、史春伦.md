## 1. 项目简介（问题定义与价值）

### 1.1 谁在什么情况下遇到什么问题？

在真实城市道路中，驾驶主体（人类驾驶员、自动驾驶车辆）会在以下场景中遇到“交通规则临时切换”的问题：

- **信号灯故障/断电**：路口不再由固定信号控制，交通组织依赖现场交警临时指挥。
- **大型活动与临时管制**：道路封控、分流、潮汐车道调整需要人工指挥实时引导。
- **交通事故处置/道路施工**：车道占用、改道与单侧放行常由交警手势进行动态调度。

在这些情况下，交警的手势（如“停止”“直行”“左转待转”等）成为交通参与者必须遵从的高优先级指令。人类驾驶员可以凭经验快速理解并执行，但**当前多数无人驾驶系统仍难以稳定理解交警的动态手势意图**，从而存在误判、违规通行、指令抖动等风险，限制无人车在开放道路的全天候、全场景适用性。

### 1.2 为什么用视觉方法去解决？

交警手势属于典型的“视觉指令”，其核心信息主要体现在：

- 手臂/上肢的空间姿态与运动方向；
- 手势序列的时间结构（开始—保持—结束）；
- 场景语义（路口、车道、指挥方向）对意图的约束。

因此，视觉感知是最直接的解决路径：通过摄像头获取视频流，结合姿态估计/手势分类/时序意图解析等视觉技术，将“人的动作”转换为“机器可执行的控制意图”。

### 1.3 已经有什么解决方法？

目前相关方向常见路线包括：

- **纯分类**：对单帧图片做手势分类，工程简单但易受噪声影响，输出抖动明显。
- **检测+分类**：先定位人体/手臂区域再分类，能提升背景鲁棒性，但仍缺乏时序理解。
- **姿态关键点驱动**：用人体关键点（骨架）表达上肢动作，结构先验更强，但在遮挡/远距离时容易丢点。
- **动作识别/时序模型**：引入滑动窗口、RNN/Transformer 等进行序列建模，性能更强但常带来算力与延迟压力。

### 1.4 我们的方案优势与用户价值

TrafficCopilot 的定位是无人驾驶平台可集成的“独立感知模块”，强调工程可用性与安全性。核心优势体现在：

- **闭环输出**：不仅输出手势类别，还输出稳定的“车辆控制域可理解指令”（含置信度与状态），贴近实际系统接口需求。
- **鲁棒性导向**：建立可配置的数据增强与评估体系，支持对雨雾/模糊/压缩/遮挡等退化场景进行压力测试。
- **时序抗抖动**：通过滑动窗口融合 + 抑制策略（hold/cooldown）稳定输出，降低单帧误判引起的指令抖动。
- **轻量可部署**：完成 PyTorch→ONNX 的导出脚本，并尝试 ONNX 动态量化，为后续边缘端部署做准备。

对用户（无人驾驶车辆/示范区运营方/安全评测方）的直接价值：

- 在信号灯失效、临时管控等场景下提升无人车遵从人工指挥的能力；
- 减少误判导致的违规通行与安全隐患，提升公众信任；
- 形成可复用的软件模块与评测方法，便于规模化推广与迭代。

## 2. 项目进度

### 2.1 技术路线图

实践周期间我们以“能运行、能训练、能展示、可复现”为目标，先跑通训练与推理闭环，再在鲁棒性与可解释性上增强。

```text
离线训练/实验
  数据集（图片分类）
     │
     ▼
  数据增强（torchvision / albumentations，可配置）
     │
     ▼
  轻量分类模型（MobileNetV3 baseline）
     │
     ├─ 导出：PyTorch → ONNX（含 dynamic_axes）
     └─ 量化：ONNX 动态量化（可选）

在线推理/演示
  图片输入（上传/样例序列）
     │
     ├───────────────┐
     ▼               ▼
  单帧分类（Top-k + 置信度门控）  骨架关键点（MediaPipe，可视化/特征）
     │               │
     └───────┬───────┘
             ▼
  时序意图引擎（滑窗加权 + hold/cooldown 抗抖动）
             │
             ▼
  指令输出（标准化枚举 + 解释 + 可追溯原因）
```

### 2.2 技术栈说明（实践周已落地部分）

- 语言与运行环境：Python 3.8+
- 深度学习：PyTorch、torchvision
- 数据增强：torchvision transforms、Albumentations（通过配置切换）
- 姿态关键点：MediaPipe（离线提取与 Demo 叠加渲染）
- 部署准备：ONNX、ONNX Runtime（导出与量化）
- Demo：Flask + 简单前端（上传图片推理、样例序列演示）

依赖清单统一记录在：`requirements.txt`。

### 2.3 数据来源与预处理方案

#### 2.3.1 当前已引入的数据

仓库已包含一份 Roboflow 导出的多分类手势数据集（用于训练/评估与 Demo 演示）：

- 路径：`demo/traffic gestures.v1i.multiclass/`
- 标注格式：Multi-Class Classification（每张图片一个类别）
- 类别（8 类）：change lanes、go straight、pull over、slow down、stop、turn left、turn right、wait for left turn
- 数据规模（由训练产物统计）：train/valid/test = 234/22/11（共 267）

#### 2.3.2 预处理与增强（已实现）

为提升泛化与鲁棒性，训练阶段实现了可配置的增强流水线：

- 通用增强：随机裁剪、翻转、轻微旋转、颜色扰动、归一化（ImageNet mean/std）
- 退化增强：模糊/压缩/遮挡等
- 天气增强（Albumentations 配方）：雨/雾/雪等退化模拟

增强后端通过配置 `augment.backend` 在 `torchvision` 与 `albumentations` 之间切换。

### 2.4 实验结果（阶段性）

仓库已落盘三组可复现实验（配置、日志、指标、权重齐全），可用于汇报“算法优化组任务：跑出初步性能数据”。

### 2.5 OpenCV 函数使用清单（已实现）

为满足“后续需自行处理原始图片与视频”的工程需求，我们已完成基于 OpenCV 的数据处理与可视化工具链，用于：原始图片/视频读写、预处理、ROI 裁剪与对齐、质量评估、调试叠加与结果导出。实践周阶段实际使用的关键函数如下（按功能归类）：

- 图片/视频 I/O：`cv2.VideoCapture`、`cv2.VideoWriter`、`cv2.VideoWriter_fourcc`、`cv2.imread`、`cv2.imwrite`
- 颜色与通道：`cv2.cvtColor`、`cv2.split`、`cv2.merge`
- 几何变换与对齐：`cv2.resize`、`cv2.getRotationMatrix2D`、`cv2.warpAffine`
- 图像增强与去噪：`cv2.GaussianBlur`、`cv2.medianBlur`、`cv2.convertScaleAbs`、`cv2.addWeighted`
- 边缘与形态学（辅助定位/调试）：`cv2.Canny`、`cv2.morphologyEx`、`cv2.getStructuringElement`
- 图像质量评估：`cv2.Laplacian`（用于模糊度估计/筛除低质量帧）
- 可视化叠加与调试：`cv2.rectangle`、`cv2.putText`、`cv2.line`、`cv2.circle`、`cv2.polylines`

说明：上述 OpenCV 工具链用于“数据层与可视化层”的工程补齐，与训练/推理模块解耦；后续接入真实路口视频流时可直接复用，并将输出统一交给分类模型与时序意图引擎。

## 4. 遇到的问题与分析

### 4.1 数据集规模小，结果易偏乐观

当前数据总量 267，test 仅 11 张；即使取得较高准确率，也可能来自“同源分布 + 样本少”带来的偶然性与过拟合风险。需要用更大规模、更真实的交警场景数据验证。

### 4.2 标签体系与真实交通语义存在差距

Roboflow 数据集的 8 类属于“手势图像分类”定义，而真实无人车控制意图可能需要引入更多状态（例如：安全降级、保持等待、缓慢通行、指令取消/结束等）。后续需要将类别体系与控制语义对齐，并明确映射表与降级策略。

### 4.3 文档与实现存在局部不一致

由于迭代速度快，部分说明文档与当前实现存在出入（例如 Demo 技术路线的描述与实际推理方案）。后续需要建立“文档随代码同步更新”的规范，避免交付与验收误差。

## 5. 待开发与解决方案（下一步计划）

### 5.1 数据与评估闭环（短期优先级最高）

- 扩充真实场景数据：夜间/雨雾/远距离/遮挡/复杂背景，建立更可信的 test 集。
- 明确类别→控制意图映射：增加 `UNKNOWN/SAFE_STOP` 等安全降级状态，并形成可追溯规则。
- 强化评估：除 acc 外增加混淆矩阵、宏平均 F1、鲁棒性保持率、指令稳定性指标（切换频率/误触发率）。

### 5.2 模型与时序（中期）

- 在更严格划分下复验：baseline、天气增强、融合模型的消融对比。
- 引入视频级验证：从“单帧分类”升级到“连续帧输入”，评估意图引擎的稳定性与延迟。

### 5.3 部署与接口（中期到长期）

- 完成 ONNXRuntime 推理对齐测试与性能统计，必要时引入 TensorRT。
- 定义 ROS2 接口与消息契约，形成可直接接入无人车软件栈的模块化输出。
